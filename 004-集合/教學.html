A slice：可變長度的清單（電商最常用的容器）

對應檔案：sample-01-slice.go ｜ 跑：go run 004-集合/sample-01-slice.go
核心觀念slice（切片）是 Go 的「可變長度清單」，對應其他語言的 List / ArrayList / Python 的 list。電商裡的購物車、訂單明細、商品列表幾乎都是 slice。它的長度會變、要幾個放幾個。

1. 用字面值建 slice

最常見的起手式是 slice 字面值 []T{ ... }。[] 裡不寫長度（寫了長度就變成下面第 4 節的 array 了），要幾個元素就給幾個：

cart := []Product{
    {Name: "機械鍵盤", Price: 2490},
    {Name: "滑鼠墊",   Price: 290},
    {Name: "USB-C 線", Price: 190},
}
// len(cart) == 3

2. append —— 往尾巴加東西（slice 的靈魂）

用內建的 append 往 slice 尾巴加元素，可以一次加多個。務必把回傳值指回原變數：

cart = append(cart, Product{Name: "手機支架", Price: 350})
cart = append(cart, a, b)   // 一次加多個
陷阱：一定要 cart = append(...)當底層陣列容量不夠，append 會配一塊更大的新陣列、回傳一個新的 slice。若寫成 append(cart, x) 卻沒把結果接回 cart，那筆就白加了（甚至改到別人）。所以慣例永遠是 s = append(s, ...)。

3. 走訪與加總 —— for range 是標配

走訪 slice 幾乎都用 for range，每輪給 (索引, 元素副本)；用不到索引就用 _：

total := 0
for _, p := range cart {
    total += p.Price
}

4. 索引 / slicing / 與 array 的差別

索引從 0 起：cart[0] 是第一項。slice 表達式 s[low:high] 取「low 到 high-1」這一段（含頭不含尾）；省略 low 預設 0、省略 high 預設 len：

top3 := cart[:3]   // 前 3 項（索引 0,1,2）

對照 array（陣列）：長度寫死在型別裡、是型別的一部分、不能變。實務上幾乎只用 slice，array 少見（認得它、知道差在哪即可）：

var q [3]int   // 固定 3 格的 array；[3]int 和 [4]int 是不同型別
q[0] = 12000
慣例「數量會變」用 slice（[]T），這是 99% 的情況；只有「數量鐵定固定」（如 RGB 三色、一週七天）才考慮 array。len(s) 拿長度、cap(s) 拿容量（底層陣列能裝多少，塞滿了 append 才會重配）。
類比slice 像「可伸縮的伸縮桿」，array 像「固定長度的鐵尺」。slice 其實是「看著一塊底層陣列的視窗」——這個觀念在群組 C 的 aliasing 陷阱會變得很關鍵，先記著。

B map：鍵值字典，與 v, ok := 判存在

對應檔案：sample-02-map.go ｜ 跑：go run 004-集合/sample-02-map.go
核心觀念map[K]V 是 Go 的字典（其他語言的 dict / HashMap），做「以 key 查 value」。電商裡的庫存表（SKU→數量）、價目表（ID→價格）、購物車數量都是它。

5. 建 map 與讀寫

型別是 map[鍵型別]值型別。可以用字面值建，之後用 key 讀寫；對「不存在的 key」賦值就是新增一筆：

stock := map[string]int{"A001": 12, "A003": 7}
stock["A001"] -= 3     // 出貨扣庫存
stock["A009"] = 20     // 對新 key 賦值 = 新增
len(stock)             // 目前有幾個 key

6. 零值陷阱 & v, ok :=（map 最重要的 idiom）

讀一個不存在的 key 不會報錯，會回傳「值型別的零值」（int 是 0、string 是 ""）。所以光看 stock["X"] == 0 分不出「庫存剛好是 0」還是「根本沒這個 key」。正解是逗號 ok 寫法：

if qty, ok := stock["A002"]; ok {
    // ok==true：key 存在，qty 是它的值（可能剛好是 0）
} else {
    // ok==false：key 不存在
}
為什麼重要「查存在」在電商 code 天天用：判斷某 SKU 在不在庫存表、某會員在不在名單、某折扣碼有沒有登記。記住口訣：要判斷 key 在不在，就用 v, ok := m[k]，看 ok，別只看值。

7. delete 刪 key & 走訪 map

delete(m, key) 刪一筆（下架商品）；刪一個不存在的 key 也不會報錯（安全）。走訪一樣用 for range，每輪給 (鍵, 值)：

delete(stock, "A002")
for sku, qty := range stock {
    fmt.Println(sku, qty)
}
陷阱：走訪順序不保證Go 的 map 走訪順序是隨機的、每次可能不同（語言故意這樣設計，防你依賴順序）。需要固定輸出順序，得先把 key 收進一個 slice、sort 排序後，再照排好的 key 去查 map。

C nil 行為與 slice aliasing（兩個經典大坑）

對應檔案：sample-03-nil與aliasing.go ｜ 跑：go run 004-集合/sample-03-nil與aliasing.go

8. nil slice —— 可以直接 append

只宣告不初始化的 slice 是 nil（len=0, cap=0），但對它 append 完全合法。要「逐步收集」一個清單時，慣例上就直接宣告再一路 append：

var items []string          // nil slice
items = append(items, "鍵盤") // 對 nil slice append，OK！
慣例需要「先空著、之後才慢慢加」的清單，直接 var xs []T 就好，不必特地 make。這也是為什麼函式回傳「空清單」時常直接回一個 nil slice——呼叫端照樣能 range、能 append、len 是 0，行為一致。

9. nil map —— 一「寫入」就 panic！

對照 slice，map 就危險多了：只宣告的 map 是 nil，讀 nil map 不會 panic（回零值），但寫入 nil map 會直接 runtime panic。map 一定要先 make（或用字面值建好）才寫：

var m map[string]int   // nil map
_ = m["x"]             // 讀：OK，回零值 0
m["x"] = 1             // 寫：💥 panic: assignment to entry in nil map

m = make(map[string]int) // 先 make
m["x"] = 1               // 現在寫入 OK
必記陷阱nil slice 可 append，nil map 寫入 panic——這是 004 最容易考倒人的對比。實務常見於：struct 裡有個 map 欄位忘了初始化，第一次寫就炸。解法是建立 struct 時就把 map make 好，或用字面值 map[K]V{}。

10. slice aliasing —— 兩個 slice 共享同一塊底層陣列

回到第 4 節說的「slice 是看著底層陣列的視窗」。把一個 slice 指派給另一個、或用 s[low:high] 切一段，得到的是「看同一塊底層陣列的另一個視窗」，不是複製資料。所以改一邊會動到另一邊：

cart  := []string{"鍵盤", "滑鼠", "螢幕", "耳機"}
promo := cart[0:2]      // promo 和 cart 共享底層陣列
promo[0] = "換一個"      // cart[0] 也跟著變成 "換一個"！

想要「真正獨立的一份」，用內建 copy 複製到新 slice：

indep := make([]string, len(cart))
copy(indep, cart)      // 複製資料；indep 和 cart 從此各走各的
indep[0] = "改這個"     // 不會影響 cart
陷阱：切子清單交出去要小心「切一段 slice 傳給別的函式去改」是電商 code 常見寫法，但它們共享底層陣列，被改到原本那份很難查。要交出「可以自由改的副本」，就 make + copy（或 append([]T(nil), s...)）。這牽涉到底層指標，主題 006 指標會講更透。
類比slice 像「同一份 Google 文件的兩個分享連結」——你以為各拿一份，其實編輯任一個，另一個也變。要真正各自一份，得「另存新檔」（copy），不是「複製連結」（指派/切片）。
