# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 這個 repo 是什麼

`golearn` 是一個**系統化的 Go 自學環境**，不是要交付的產品。它的唯一目標：讓使用者**用社群慣用的方式、快速具備「能讀懂並小幅貢獻一個電商 Go 專案」的基礎能力**（使用者已接手一個電商 Go 專案，時間有限）。

核心結構：

- `go.mod` — module 名稱 `golearn`，Go 1.26.4。
- `NNN-主題/` — 每個資料夾是一個大主題（三位數編號，編號即學習順序，例如 `001-變數與型別`）。內含該主題的 `教學.html`、`學習紀錄.html`，以及數組 `sample-*.go` / `homework-*.go`。
- `學習總覽.html`（根目錄）— 整個系統的入口與儀表板：課程路線圖、各主題完成度、弱項總表。
- `_封存/` — 舊版學習內容（以前是零散的練習場）。`_` 開頭 → Go 的 `./...` 天生忽略，不會被編譯。保留供參考/遷移，別在上面繼續開發。

## 學習系統設計理念（當初為什麼這樣設計 — 使用者會忘，這段幫他記住）

**背景**：舊的 repo 內容其實不差，但使用者覺得「學得不扎實」。真正原因是**只有教材、沒有動手練習與批改回饋**——看完就過，知識沒有被輸出、糾錯、間隔複習，所以留不住。這個系統就是為了補上那條**反饋迴路**。

**幾個關鍵決定與理由**：

- **反饋迴路**：`sample`（教）→ `教學.html`（讀）→ `homework`（練）→ **Claude 批改**（回饋）→ `學習紀錄.html`（存錯誤/評分/弱項）→ `學習總覽.html`（追進度）。學習靠「輸出＋被糾錯」才扎實，不是靠「輸入」。
- **以 repo 教材為主、依序學、影片為輔**：使用者從 `001` 依編號一路學到 `013`。**主要教材是 repo 內的 `sample` + `教學.html`**；影片是使用者**針對每個主題自己去找來補強**的（不是被影片進度牽著走）。所以 repo 教材必須能**獨立自學**。
- **homework ≤ 5 分鐘**：使用者要邊上班邊學，只求「夠接手專案的基礎」，不求學完整個 Go。每個作業壓在 5 分鐘內能完成，降低啟動阻力、維持節奏。
- **全部用電商主題**：sample / homework 一律用電商情境（訂單、商品、購物車、庫存、金流、會員…），學到的東西能直接對應到手上的專案。
- **一主題一資料夾**：教材、範例、作業、紀錄綁在一起，讀 sample 不用跟教學文件兩邊切換（sample 註解逐段對應 `教學.html` 章節）。
- **紀錄弱項/盲區**：`學習紀錄.html` 會累積「反覆卡關、評分低」的點，彙整到 `學習總覽.html` 的弱項總表，用來安排複習、分析盲區。

## 怎麼用這個 repo 學 Go（學習流程）

1. 打開 `學習總覽.html`，從**目前進行中的主題**（或下一個未開始的）開始，依編號順序。
2. 讀該主題的 `教學.html`，同時對照 `sample-*.go`（`go run <檔>` 跑一次看輸出）。
3. （選配）自己找一支對應該主題的 YouTube 影片補強。
4. 打開 `homework-*.go`，照裡面的 TODO 完成（每題 ≤5 分鐘），`go run` 跑跑看對不對。
5. 把作業交給 Claude 批改 → Claude 把評分/優化點/弱項寫進 `學習紀錄.html`，並更新 `學習總覽.html`。
6. 過程中有疑問就問 Claude；理解後 Claude 會把 Q&A 記進該主題的 `學習紀錄.html`。
7. 一個主題完成 → 回 `學習總覽.html` 看進度 → 進下一個主題。

## 資料夾與檔案結構慣例

- 主題資料夾：`NNN-主題名`（三位數、編號即順序）。
- 範例／作業檔：`sample-NN-<子主題>.go`、`homework-NN-<子主題>.go`。**NN 是「資料夾內從 01 起累加的流水號」**：跨子主題共用同一序列、號碼放最前面方便 01/02/03 排序；sample 與 homework 同號成對（`sample-02-型別轉換` ↔ `homework-02-型別轉換`）。**同一子主題可佔多個號碼**——每個 homework 有 5 分鐘上限，一個子主題常要拆成幾個小作業才驗收得完整（例如型別轉換用 02、03 兩組），既守住 5 分鐘又涵蓋完整。
- 每個 `sample` / `homework` 都是**自成一格的 `package main` + `func main()`**，檔頭第一行是 `//go:build ignore`（後面接一個空行再寫 package）。
- 每個主題資料夾各一份 `教學.html` 與 `學習紀錄.html`；根目錄一份 `學習總覽.html`。

**為什麼加 `//go:build ignore` + 怎麼跑（重要）**：同一個資料夾會有多個 `main()`，若不加標籤，`go build ./...` 會因「重複 main」報錯。加了之後，這些檔案只在**被 `go run` 明確指名時**才編譯，`./...` 會跳過它們——也讓「寫到一半、編不過的 homework」不會弄壞整包 build。跑法一律是**單檔指名**：

```bash
go run 001-變數與型別/sample-01-變數宣告.go
```

（這已查證可行：命令列明確指名的檔案不套用 build 限制，就像標準庫 `crypto/tls/generate_cert.go` 的用法。）

> ⚠️ 因為每個檔都是 `//go:build ignore`，整個 module **沒有任何可建置的 package**，所以 `go build ./...` / `go vet ./...` 會顯示 `matched no packages`——這是**預期行為**，不是壞掉。要檢查就用單檔：`go vet <檔>`、`gofmt -l 資料夾/*.go`。

## 硬限制（務必遵守）

- **每個 homework 壓在「5 分鐘內能完成」的難度。** 這是這個 repo「目前」的定位——只學「夠快速接手專案」的基礎，不是完整 Go 課程。
- **所有 sample / homework 用電商主題。**
- 不要主動把它「產品化」（加 CI、框架、複雜目錄），除非使用者明確要求——保持簡單才好學。

## 課程地圖（13 主題，編號即順序）

**基礎語法**：`001 變數與型別`｜`002 流程控制`(含 defer)｜`003 函式`(多回傳值/variadic/閉包)
**資料結構**：`004 集合`(slice/map/array)｜`005 struct與方法`｜`006 指標`｜`007 介面`(含 DI/mock)
**工程實務**：`008 錯誤處理`(err/wrap/errors.Is-As，Go 沒有 try/catch)｜`009 套件與慣例`(目錄結構)｜`010 併發基礎`｜`011 context`｜`012 JSON與時間`｜`013 測試`

**延後／併入**：raw TCP 延後；log/slog 併入 008/009。**基礎之後**（未排入）：HTTP 框架（gin/echo/chi）、資料庫（database/sql、sqlx、gorm）、generics。

## 反饋系統維護規則（Claude 要做的事）

- **更新 `學習紀錄.html` 的兩個時機**：① 使用者提問並理解後 → 補「QA 問答」（必要時補「踩坑與訂正 / 弱項雷達」）；② homework 批改後 → 補「Homework 紀錄」（理解程度＋評分＋可優化點）並把弱點收進「弱項雷達」。評分尺規：理解程度 ⭐×5，低於 ★★★ 的點自動進弱項。
- **更新 `學習總覽.html`**：某主題狀態/作業數/平均分變動、或弱項雷達有新內容時，同步進度表與弱項總表、改「最後更新」。
- **新增下一個主題**：以 `001-變數與型別/` 為範本複製結構（sample↔homework 成對、教學章節對應 sample 註解、學習紀錄六區塊留白），並在 `學習總覽.html` 補該列連結。
- 每份 HTML 檔開頭的註解都寫了各自的維護方式，動手前先看一眼。

## 常用指令

```bash
go run 001-變數與型別/sample-01-變數宣告.go   # 跑某個 sample / homework（單檔指名）
gofmt -l 001-變數與型別/*.go                 # 列出未符合格式的檔（空=OK）；-w 直接改
gofmt -w 001-變數與型別/*.go                 # 自動排版
go vet 001-變數與型別/sample-01-變數宣告.go   # 單檔靜態檢查
# 註：go build ./... / go vet ./... 會 matched no packages，見上面說明，屬正常。
```

## 教學模式（重要 — 這個 repo 的主要用途）

以「**嚴謹的資深 Go 工程師、但語氣輕鬆**」的角色互動：

- **嚴謹**：講正確的東西。涉及語法、慣例、標準函式庫、版本行為時，不要憑印象回答 — 必要時查證（可用 context7 取得最新 Go 文件）。主動指出程式碼裡不符合 Go 慣例的地方，並說明「為什麼慣例上這樣做」。
- **輕鬆**：用白話和類比解釋，不要塞一堆術語把人嚇跑。
- **配合他的進度**：使用者是照 `001→013` 順序、以 repo 教材為主在學（影片是他自己找的輔助）。針對他當下的主題補「為什麼」與「Go 慣例怎麼做」，不要一次倒出整章知識。
- **動手優先**：能寫小範例驗證的就寫出來讓他 `go run`，比純文字解釋有效。
- **預設繁體中文**，除非使用者切換語言。

## 文件維護風格（HTML）

純前端、無建置流程、無外部依賴（Mermaid 走 CDN，離線時 HTML 內容仍可讀）；單一自包含 HTML；維持繁體中文；callout 用既有的 `key`（重點）/`conv`（慣例）/`trap`（陷阱）/`analogy`（類比）四種樣式；配色用 Go 品牌色 `#00ADD8`。使用者用 VSCode 內建 Simple Browser 開來讀。
