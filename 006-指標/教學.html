A 傳值 vs 指標：誰改得到本尊？

對應檔案：sample-01-傳值與指標.go ｜ 跑：go run 006-指標/sample-01-傳值與指標.go
整個主題的核心一句話Go 永遠是「傳值（pass by value）」——傳進函式的都是一份 copy。想改到呼叫端的本尊，就把「地址」傳進去，讓兩邊指向同一塊記憶體。

1. 傳值心法：函式收到的是「一份 copy」

把一個 Order 傳進函式，函式收到的 o 只是複製品。函式裡怎麼改 o，都只動到 copy；函式 return 後 copy 被丟掉，原本的訂單毫髮無傷。

func addItemByValue(o Order, p Product) {
    o.Items = append(o.Items, p)   // 改的是 copy
    o.Status = "updated"
}
// 呼叫端：orderA 完全沒變，件數還是 0
addItemByValue(orderA, keyboard)
類比像把「文件影印一份」交給同事改。他在影本上塗改得再認真，你手上的正本一個字都不會變。要他改到正本，你得把正本的「所在位置（地址）」告訴他。

2. 傳指標：用 & 取地址，函式透過地址寫回本尊

把參數型別改成 *Order（指向 Order 的指標），呼叫時用 & 取地址傳進去。函式透過這個地址寫入，動到的就是本尊。

func addItemByPointer(o *Order, p Product) {
    o.Items = append(o.Items, p)   // o.Items 是 (*o).Items 的縮寫
    o.Status = "updated"
}
addItemByPointer(&orderB, keyboard)   // & 取地址 → orderB 真的被改
慣例對指標取欄位可直接寫 o.Items，不用寫成 (*o).Items——Go 會自動幫你解參考(dereference)。這就是為什麼電商 code 裡 *Order 的方法裡到處是 o.欄位，看起來和值一樣。

3. 三個符號速記：*T 型別、&x 取地址、*p 解參考

指標只有三個符號，別被 * 一符兩用搞混——看它出現在哪：

符號出現位置意思
*T型別位置，如 var p *int「指向 T 的指標」這個型別
&x值前面，如 &price取變數 x 的地址（得到一個指標）
*p指標前面，如 *p = 250解參考：透過指標讀/寫它指向的本尊
 
price := 100
var p *int = &price   // *int 是型別；&price 取地址
fmt.Println(*p)        // *p 解參考，讀出 100
*p = 250               // *p 解參考，寫回本尊 → price 變 250

4. 「傳指標」其實也是傳值：copy 的是「地址」

別把指標理解成「就不 copy 了」。指標本身也是一個值（一串地址數字），傳進函式一樣被複製一份；只是這份 copy 和原指標指向同一塊記憶體，所以透過它寫入才動得到本尊。sample 裡印出「函式內收到的地址」和「外面的地址」完全一樣，就是證明。

為什麼要懂這個因為它解釋了一個常見疑惑：既然傳指標也 copy，為什麼就改得到？因為 copy 的是「地址」不是「資料」——兩份地址指向同一筆訂單。這也預告了主題 010 併發：多個地方共享同一塊記憶體，才需要 Mutex 保護。

B pointer receiver：方法要改到自己就用 *T

對應檔案：sample-02-pointer接收者.go ｜ 跑：go run 006-指標/sample-02-pointer接收者.go

這是主題 005「方法 / receiver」的續集：現在補上「receiver 該用值 (o Order) 還是指標 (o *Order)」。判斷法則其實就是第 1~2 節的心法搬到方法上。

5. 要「改到自身」的方法 → pointer receiver (o *Order)

方法本質上也是「把 receiver 當參數傳進去」。所以 value receiver 收到的是訂單的 copy，改了白改；要真的改到訂單，receiver 得寫成 *Order。

func (o *Order) ApplyDiscount(percent int) {
    for i := range o.Items {
        o.Items[i].Price = o.Items[i].Price * (100 - percent) / 100
    }
}
func (o *Order) AddItem(p Product) { o.Items = append(o.Items, p) }
Go 的貼心對「可定址(addressable)」的變數（例如一般區域變數 order）呼叫指標方法時，Go 會自動幫你取地址：order.ApplyDiscount() 等於 (&order).ApplyDiscount()。所以你不用每次都手動寫 &，用起來和值方法一樣自然。

6. 只「讀」不「改」→ value receiver (o Order)

Total() 只是加總、不動任何欄位，用 value receiver 就好——語意清楚（「我保證不改你」）、也不怕 nil。

func (o Order) Total() int {
    sum := 0
    for _, item := range o.Items { sum += item.Price }
    return sum
}
陷阱：別混用同一個型別的方法，慣例是「全值」或「全指標」二選一，不要混。混用會讓「這型別到底能不能被指標方法改」變得混亂，也可能踩到「值 vs 指標」是否滿足介面的雷（主題 007 會遇到）。實務上只要有任何一個方法需要改自身，就全部用 pointer receiver。

7. 陷阱回顧：range 的元素是「副本」，直接改沒用

這點在主題 004 提過，這裡用「值/指標」再解釋一次為什麼：for _, item := range o.Items 每輪的 item 是元素的copy，改 item.Price 只動到 copy。要改原值，得用索引直接寫回 slice：

// ❌ 白改：item 是副本
for _, item := range o.Items { item.Price = 0 }

// ✅ 用索引改到本尊
for i := range o.Items { o.Items[i].Price = 0 }
一句話收range 給的是副本 → 要改原值就用 xs[i] 索引寫回。這和第 1 節「傳值改不到本尊」是同一個道理的不同長相。

C new 與 make：兩種「配置記憶體」的內建函式

對應檔案：sample-03-new與make.go ｜ 跑：go run 006-指標/sample-03-new與make.go

Go 有兩個內建函式在「配置記憶體」，新手常搞混。一句話分：new 回指標、make 回本體；而且 make 只給 slice / map / channel 用。

8. new(T)：配置一塊「零值」記憶體，回傳 *T

new(Product) 配置一個零值的 Product（{Name:"", Price:0}），回傳指向它的指標 *Product。任何型別都能 new。

p1 := new(Product)   // p1 是 *Product，指向零值
p1.Name = "4K 螢幕"   // p1.Name 是 (*p1).Name 的縮寫
p1.Price = 8990

9. struct 慣例：直接用 &T{...}（幾乎取代 new）

&Product{...} 一樣拿到 *Product，但能一行帶初值，所以 Go 程式裡幾乎都用 &T{}、很少寫 new。兩者拿到的東西型別相同。

p2 := &Product{Name: "喇叭", Price: 1200}   // 型別也是 *Product，還能給初值
慣例建 struct 用 &T{...}；new 在真實 Go code 幾乎見不到（頂多用在「想要一個某型別的零值指標、又沒有欄位要填」的少數場合）。看到別人寫 &Order{} 就是這裡的正解。

10. make：只給 slice / map / channel，回傳「本體」不是指標

make 專門用來建需要初始化內部結構的三種型別：slice、map、channel。它回傳的是可以直接用的本體（不是指標）。

cart  := make([]Product, 0, 4)   // slice：len=0, 預留 cap=4
stock := make(map[string]int)    // map：建好才能寫入
用在回傳典型寫法
new(T)任何型別*T（指標）少用；struct 改用 &T{}
&T{}struct*T（指標）&Order{ID: 1}
makeslice / map / channel本體（非指標）make(map[string]int)

11. 陷阱：nil map「讀」可以、「寫」會 panic

只宣告沒 make 的 map 是 nil（呼應主題 004）。讀 nil map 不會炸（回零值），但寫入會 panic。所以要寫入前一定先 make（或用字面值 map[string]int{}）。

var priceTable map[string]int      // nil map
_ = priceTable["不存在"]            // 讀 OK，得到零值 0
// priceTable["鍵盤"] = 2490       // ← 寫入會 panic: assignment to entry in nil map
priceTable = make(map[string]int)  // 先 make
priceTable["鍵盤"] = 2490           // 現在才能安全寫入
陷阱nil slice 和 nil map 待遇不同：nil slice 可以 append（append 會幫你配空間），但 nil map 一寫入就 panic。所以拿到 map 一律先 make。（slice 的 nil 行為主題 004 已詳談。）
