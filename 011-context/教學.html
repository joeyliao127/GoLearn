主題 011 · context

context.Context 是電商後端的「請求生命線」—— 取消、逾時、帶著 request 範圍的值一路往下傳。看懂它，你就看懂了為什麼幾乎每個 web handler、每個 DB 查詢的第一個參數都是 ctx。

你會學到為什麼電商 code 到處都是 ctx：它讓一條呼叫鏈能被「統一取消／設逾時」，還能沿路攜帶像 requestID 這種 request 範圍的值。三個 sample 分別對應「傳遞」「逾時與取消」「帶值」。
怎麼用這份教材每節標題下方會標「對應哪個 sample 檔」。先讀說明 → 打開該 sample 對照註解 → go run 跑一次看輸出 → 最後做動手練習。
先有個畫面把 ctx 想成餐廳的「這一桌的服務鈴＋點餐單」：整桌（一個請求）共用一個鈴，經理一按（cancel／逾時），從前台到廚房全都收到「這桌不做了」；點餐單上還夾著桌號（requestID），每個經手的人都看得到。

A 傳遞：為何要 context、ctx 當第一參數、根 context

對應檔案：sample-01-傳遞.go ｜ 跑：go run 011-context/sample-01-傳遞.go

1. 為什麼需要 context

想像一個電商請求進來：handler → service → repository → DB。這條鏈上如果顧客關掉頁面、或已經等太久，我們會希望「整條鏈一起停下來」，別再白白佔用 DB 連線。可是這些函式各自獨立，怎麼一起收到「停」的訊號？

答案就是 context.Context：它是一個沿著呼叫鏈往下傳的物件，攜帶三種 request 範圍的東西——

  • 取消訊號：上游一喊停，下游全收到。
  • 逾時 / deadline：超過時間自動視為取消。
  • request 範圍的值：像 requestID、登入者身分（第 8~10 節細講）。
為什麼重要電商後端幾乎每個「會 I/O、會花時間」的函式（HTTP handler、DB 查詢、呼叫外部 API）都吃 ctx。看懂 context，你才讀得懂這些函式簽章為什麼長那樣，也才寫得出「可以被取消／逾時」的正確程式。

2. ctx 當「第一個參數」，型別 context.Context，命名叫 ctx

這是 Go 鐵一般的慣例：只要函式可能要被取消／逾時／讀 request 範圍值，就把 ctx 放參數列最前面，型別寫 context.Context，變數就叫 ctx。

// 慣例長相：ctx 一定在最前面
func handleGetOrder(ctx context.Context, orderID int) { ... }
func getOrderService(ctx context.Context, orderID int) (Order, error) { ... }
func getOrderFromDB(ctx context.Context, orderID int) (Order, error) { ... }
慣例ctx 放第一參數、命名固定叫 ctx。中間層就算自己沒用到 ctx，也有義務把它繼續往下傳，讓最底層（DB）也收得到取消／逾時訊號。
陷阱：別把 ctx 存進 struct不要把 ctx 當欄位塞進 struct 長期保存。ctx 是「跟著一次請求走」的、生命週期很短；存進 struct 會讓它活太久、語意也亂掉。慣例就是「用參數傳」，每次呼叫顯式帶著它。

3. 根 context：Background()

最上層要先有一個「根」ctx。慣例：main、程式初始化、測試、以及「剛收到一個請求」的地方，都用 context.Background()——它永不取消、沒有值、沒有逾時，是一切 ctx 的起點，之後才在它上面「派生」出帶逾時或帶值的子 ctx。

ctx := context.Background()   // 根 ctx（真的 web 框架會幫你準備好）
handleGetOrder(ctx, 1001)     // 往下傳
Background vs TODO還有一個 context.TODO()，長得跟 Background() 一樣，語意是「我還不確定該用哪個 ctx，先佔位」。正式的起點用 Background()；只有在「這裡以後應該要有真正的 ctx、但我暫時還沒接上」時才用 TODO()。

B 逾時與取消：WithTimeout / WithCancel、select、defer cancel

對應檔案：sample-02-逾時與取消.go ｜ 跑：go run 011-context/sample-02-逾時與取消.go

情境：結帳時要呼叫「外部金流閘道」授權刷卡。這種外部呼叫可能很慢，我們不能無限等——給它一個逾時，超過就放棄、回錯誤給顧客，別卡住整條請求。

4. WithTimeout：派生一個「會自己逾時的 ctx」

context.WithTimeout(parent, d) 從一個父 ctx 派生出子 ctx，並回傳兩個值：帶逾時的 ctx，加一個 cancel 函式。過了 d 這段時間，子 ctx 會自動被取消。

ctx, cancel := context.WithTimeout(context.Background(), 200*time.Millisecond)
// 200ms 後，ctx 自動進入「已取消」狀態
類比像叫外送時設一個鬧鐘：「200ms 內沒送到就別等了」。時間一到，鬧鐘（ctx）自動響，通知所有在等的人「不等了」。WithDeadline 是它的雙胞胎，差別只在「你給的是一段時間長度，還是一個明確的時間點」。

5. 用 select 等「做完」或「ctx.Done()」

怎麼「同時等操作完成、又能被逾時打斷」？用 select：一邊等操作結果的 channel，一邊等 ctx.Done()——這是一個「ctx 被取消時就會關閉」的 channel。誰先發生就走誰。

select {
case <-ctx.Done():
    // ctx 被取消或逾時了。ctx.Err() 說明原因：
    // 逾時 → context.DeadlineExceeded；被主動取消 → context.Canceled
    return ctx.Err()
case auth := <-done:
    // 授權在期限內完成
    fmt.Println("授權成功：", auth)
}
核心句型select { case <-ctx.Done(): return ctx.Err() ... } 是 context 最經典的用法。ctx.Done() 給你「何時該停」的訊號、ctx.Err() 給你「為什麼停」的原因。判斷是不是逾時可用 errors.Is(ctx.Err(), context.DeadlineExceeded)（errors.Is 來自主題 008）。

6. 黃金慣例：拿到 cancel 就立刻 defer cancel()

WithTimeout / WithCancel / WithDeadline 都會回傳一個 cancel 函式。不管你有沒有用到它、也不管會不會自己逾時，都一定要呼叫 cancel()，否則它綁的計時器／內部資源會洩漏。放在 defer 最保險：

ctx, cancel := context.WithTimeout(parent, 200*time.Millisecond)
defer cancel()   // 下一行立刻 defer，不管從哪條路徑 return 都會清理
慣例看到 WithTimeout / WithCancel，下一行幾乎一定是 defer cancel()——這和主題 002 學的「取得資源就下一行 defer 釋放」是同一套肌肉記憶。go vet 也會抓「拿了 cancel 卻沒呼叫」的漏洞。
陷阱：別忽略 cancel有人覺得「反正 WithTimeout 會自己逾時，cancel 不用管」——錯。就算會逾時，提早完成時你也該馬上 cancel 來釋放資源；defer cancel() 幫你在函式結束時一定補上這一刀。用 _ = cancel 把它丟掉是明確的 bug。

7. WithCancel：主動喊卡

WithTimeout 是「時間到自動取消」；context.WithCancel(parent) 則是「由你決定何時取消」——呼叫它回傳的 cancel()，ctx.Done() 就會關閉。適合「其中一件事失敗或不需要了，就取消其餘還在跑的工作」。

ctx, cancel := context.WithCancel(context.Background())
defer cancel()
// 風控判定可疑 → 主動喊停：
cancel()   // ctx.Done() 關閉、ctx.Err() 變 context.Canceled
類比WithTimeout 像設定的鬧鐘（時間到自己響）；WithCancel 像你手上的開關（你想關就關）。兩者常一起用：主流程設逾時，某個子任務出事時再手動 cancel 提早收工。
延伸（主題 010 銜接）主題 010 併發基礎會學到 select 搭配多個 channel、以及 goroutine。context 的 ctx.Done() 正是把「取消」變成一個 channel，讓它能自然融進 select——這是「用 context 控制一群 goroutine 生死」的基礎。

C 傳值：WithValue、未匯出 key、型別斷言、慎用

對應檔案：sample-03-傳值.go ｜ 跑：go run 011-context/sample-03-傳值.go

情境：每個進來的請求都帶一個 requestID（給 log 追蹤用），有時還有登入的會員 ID。這種「跟著這個請求走、每層都可能想讀」的值，不方便一路加進每個函式的參數列，就放進 ctx 帶著走。

8. key 用「自訂的未匯出型別」，不要用字串

context 是全程式共用的。如果大家都用字串 "requestID" 當 key，不同套件很容易「不小心用了同一個字串」而互相覆蓋。慣例是宣告一個只有本套件看得到的私有型別當 key，從型別層面保證不撞：

type ctxKey int   // 未匯出（小寫開頭），外部套件拿不到，也就無法覆蓋

const (
    keyRequestID ctxKey = iota   // 用 iota 給每個 key 一個獨一無二的值
    keyUserID
)
陷阱：別用字串或內建型別當 key寫 ctx.Value("requestID") 能動，但兩個套件用同一個字串就會互踩。go vet 會警告「不該用內建型別當 context key」。正解永遠是「自訂未匯出型別 + 常數」。

9. 用 WithValue 把值「疊」上去

context.WithValue(parent, key, val) 派生一個「帶上這組 key/value」的新 ctx。可以一層層疊很多組。慣例上會包成小 helper，讓呼叫端不用直接碰 key：

ctx = withRequestID(ctx, "req-20260701-4417")     // 包過的 helper
ctx = context.WithValue(ctx, keyUserID, 88001)    // 直接寫也行

func withRequestID(ctx context.Context, id string) context.Context {
    return context.WithValue(ctx, keyRequestID, id)
}
慣例把「塞值」和「取值」都各包一個函式（withRequestID / requestIDFrom），呼叫端只給值、看不到 key 細節，也把型別斷言集中在一處。

10. 讀值：型別斷言 + 處理「拿不到」

ctx.Value(key) 回傳 any（interface{}），要用「逗號 ok」型別斷言轉回原本型別；若這個 key 沒被設過，會拿到 nil、ok 為 false：

func requestIDFrom(ctx context.Context) string {
    if id, ok := ctx.Value(keyRequestID).(string); ok {
        return id
    }
    return "unknown"   // 拿不到就給預設值，別 panic
}
陷阱：只放 request-scoped 的東西WithValue 很好用，但別濫用成「萬用參數袋」。只放「跟著這個請求走」的資料：requestID、登入者身分、追蹤用的 trace ID。函式真正的正經輸入（訂單金額、商品清單）該用一般參數傳；像 DB 連線、logger 這種相依，該用主題 007 的介面注入——不要塞進 ctx。塞進去會讓函式簽章騙人（看不出它真正需要什麼）、還喪失編譯期檢查。
一句話記住ctx.Value 是給「橫向、跨層、跟著請求走」的中繼資料用的，不是拿來偷懶傳參數的。判準：這個值「換一個請求就會變、且沿途很多層都可能想讀」→ 適合放 ctx；否則用參數。
