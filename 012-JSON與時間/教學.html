A 序列化：struct ↔ JSON、struct tag、omitempty

對應檔案：sample-01-序列化.go ｜ 跑：go run 012-JSON與時間/sample-01-序列化.go

1. Marshal —— struct 變 JSON

把 Go 的值轉成 JSON 叫「序列化 / Marshal」。json.Marshal(v) 回傳 ([]byte, error)——注意它給的是 位元組切片，要印出來得先 string(b) 轉成文字。慣例上一定要接住那個 err。

b, err := json.Marshal(order)
if err != nil { /* 處理錯誤 */ }
fmt.Println(string(b))   // []byte 要轉 string 才是人看得懂的 JSON

// 想要有排版（縮排）的版本，方便 log / debug：
pretty, _ := json.MarshalIndent(order, "", "  ")  // (值, 前綴, 縮排)
只有大寫欄位會被序列化這是本主題最重要的一條規則：只有「大寫開頭（exported）」的欄位 encoding/json 才看得到。小寫欄位（如 internal）它完全無視，不會出現在 JSON 裡，也塞不回去。這跟 005 的「大小寫決定匯出」是同一套規則。

2. struct tag —— 控制 JSON 的鍵名

Go 欄位習慣 MixedCaps（ID、Customer），但 JSON 慣例常是小寫 id、customer。用寫在反引號裡的 struct tag json:"鍵名" 把兩邊接起來（這正是 005 學過的 tag，這裡終於派上用場）。

type Order struct {
    ID       string `json:"id"`       // JSON 鍵名變小寫 id
    Customer string `json:"customer"`
    Amount   int    `json:"amount"`   // 金額用 int，別用 float
    Note     string `json:"-"`        // "-" ＝ 永不序列化
}
tag 寫法效果
json:"id"JSON 鍵名用 id
json:"coupon,omitempty"鍵名 coupon，且值為零值時整個省略（見第 3 節）
json:"-"這個欄位永遠不序列化（例如密碼、內部備註）
沒寫 tag直接用欄位名當鍵名（大小寫照舊，如 Amount）
慣例電商專案的 struct 幾乎每個欄位都會掛 json tag，好讓 API 的 JSON 鍵名穩定、符合前端約定。看到反引號裡的 json:"..." 就是在講「這欄位在 JSON 裡叫什麼」。

3. omitempty —— 零值就從輸出消失

tag 加上 ,omitempty，當該欄位是「零值」（數字 0、字串 ""、nil、false）時，就不會出現在 JSON 裡。適合「非必填、沒有就別送」的欄位，如優惠券。

// Coupon 有值 → {"id":"A-1001",...,"coupon":"SUMMER10"}
// Coupon 是 ""  → {"id":"A-1002",...}  （沒有 coupon 這個鍵）
陷阱omitempty 判斷的是「Go 的零值」，不是「JSON 的 null」。所以 Amount int 若剛好是 0，加了 omitempty 也會被省略——金額為 0 是合法值時，別對它用 omitempty，不然接收端會以為「沒這個欄位」。

4. Unmarshal —— JSON 塞回 struct

反過來把 JSON 還原成 struct 叫「反序列化 / Unmarshal」。關鍵：第二個參數要傳指標 &order，函式才改得到你的變數（這就是 006 指標的用途）。

var order Order
if err := json.Unmarshal(data, &order); err != nil { /* ... */ }
fmt.Println(order.Amount)   // 已經是 Go 的 int，可直接運算
多的欄位會被安靜忽略JSON 裡有、但 struct 沒定義的鍵，Unmarshal 不會報錯，直接跳過。這通常是好事（前端多送欄位不會炸），但也代表打錯字的 tag 會靜靜地漏資料——想「完整接住」未知欄位，看第 11 節的 map[string]any。

B time.Time：建立、格式化、加減、時間戳

對應檔案：sample-02-時間.go ｜ 跑：go run 012-JSON與時間/sample-02-時間.go
為什麼 sample 用固定時間教學範例刻意用 time.Date(...) 造一個固定基準時間，而不是 time.Now()——這樣每次 go run 的輸出都一樣，方便你對照。實務上「現在時間」才用 time.Now()。

5. 建立一個 time.Time

時間在 Go 裡是 time.Time 型別。手動指定某個時間用 time.Date；要「現在」用 time.Now()。指定時區能讓輸出穩定、不受本機時區影響。

placedAt := time.Date(2026, time.January, 10, 9, 30, 0, 0, time.UTC)
//                 年    月         日  時 分 秒 奈秒  時區
now := time.Now()   // 真的要「現在」時才用這個

6. 格式化 / 解析 —— Go 特有的「參考時間」版面

這是 Go 最反直覺的一點：格式化不用 yyyy-MM-dd，而是拿一個固定的參考時間當範本——2006-01-02 15:04:05。你想要什麼格式，就把這個參考時間「排成那個樣子」傳給 Format。

參考數字代表
2006年
01月
02日
15時（24 小時制）
04分
05秒
 
placedAt.Format("2006/01/02 15:04")   // 2026/01/10 09:30
placedAt.Format(time.RFC3339)          // 2026-01-10T09:30:00Z（標準格式）

// 反向：字串 → time.Time，回傳 (time.Time, error)
t, err := time.Parse(time.RFC3339, "2026-01-17T09:30:00Z")
類比別的語言是給你一本「符號字典」（y=年、M=月…）要你拼；Go 是給你一張範本照片（2006-01-02…），你照著擺就好。記法：月 1、日 2、時 3、分 4、秒 5、年 6、時區 7（1 2 3 4 5 6 7 順著數）。
慣例API／DB 交換時間優先用 time.RFC3339（就是 2026-01-10T09:30:00Z 這種）。time 套件有內建常數，不用自己記版面，也跨語言通用。

7. 加減時間 —— AddDate 與 Add

兩種加減：AddDate(年,月,日) 適合「天/月/年」級距（如 7 天到期）；Add(Duration) 適合「時/分/秒」級距（如保留庫存 30 分鐘）。

expireAt  := placedAt.AddDate(0, 0, 7)      // 下單 +7 天
holdUntil := placedAt.Add(30 * time.Minute) // +30 分鐘

// 比較先後用 Before / After / Equal（別用 == 比 time.Time）
expireAt.After(placedAt)   // true
陷阱別用 == 比較兩個 time.Time——它還帶了時區與「單調時鐘」資訊，看起來一樣的兩個時間 == 可能為 false。要比先後用 Before／After，要比「同一時刻」用 Equal。

8. 兩時間相減 Sub、以及 Unix 時間戳

兩個時間相減用 a.Sub(b)，得到 Duration（一段時長），再用 .Hours() 等換算。「幾天」就是 .Hours()/24。

elapsed := t2.Sub(t1)              // Duration，例如 168h0m0s
days := elapsed.Hours() / 24       // 7

Unix 時間戳是「從 1970-01-01 UTC 起算的秒數」，一個純數字，是前後端／跨語言交換時間最通用的格式。

ts := placedAt.Unix()              // 1768037400（秒）
back := time.Unix(ts, 0).UTC()     // 時間戳 → time.Time
Duration 是有型別的數字30 * time.Minute 能寫出來，是因為 time.Minute 本身是個 Duration 常數。所以「半小時」寫 30 * time.Minute、「兩小時」寫 2 * time.Hour，讀起來像英文，也不會搞錯單位。

對應檔案：sample-03-JSON進階.go ｜ 跑：go run 012-JSON與時間/sample-03-JSON進階.go

9. 巢狀 struct 與 slice

真實訂單是巢狀的：一筆 Order 底下有 Customer（物件）和 []Item（陣列）。你不用做特別的事——只要欄位型別是別的 struct 或 slice，Marshal 會自動一路往下轉，slice 變成 JSON array、struct 變成 JSON object。

type Order struct {
    ID       string    `json:"id"`
    Customer Customer  `json:"customer"`   // 巢狀物件
    Items    []Item    `json:"items"`      // 變成 JSON 陣列
    PlacedAt time.Time `json:"placed_at"`
}

10. time.Time 進 JSON —— 自動變 RFC3339

把上面那個 PlacedAt time.Time Marshal 出來，它會自動變成 RFC3339 字串，你什麼都不用做：

"placed_at": "2026-01-10T09:30:00Z"
為什麼會自動因為 time.Time 自己實作了 MarshalJSON／UnmarshalJSON（用 RFC3339 格式）。任何型別只要實作這兩個方法，就能決定「自己怎麼變 JSON」——這是 encoding/json 的擴充點，你手上的專案可能會看到自訂型別這樣做。

11. 未知結構 —— map[string]any

收到「沒有對應 struct」或「欄位會變」的 JSON（如第三方 webhook），可以 Unmarshal 進 map[string]any（any 是 interface{} 的別名，004/007 出現過）。取值時要用型別斷言 .(型別) 還原成具體型別。

var m map[string]any
json.Unmarshal(raw, &m)

event  := m["event"].(string)    // 斷言成 string
amount := m["amount"].(float64)  // ⚠️ JSON 數字一律 float64！
tags   := m["tags"].([]any)      // 陣列 → []any
meta   := m["meta"].(map[string]any)  // 巢狀物件 → 再一層 map
最常踩的雷用 map[string]any 接時，JSON 的數字不管整數或小數，一律被解成 float64。所以金額要寫 .(float64)，寫成 .(int) 會直接 panic。要整數就自己再 int(amount) 轉一次。
慣例能定義 struct 就盡量定 struct（有型別檢查、好讀、IDE 有提示）；只有在「結構真的不固定 / 只想撈幾個欄位」時才退而用 map[string]any。別把它當預設作法。
