A testing 基礎與 table-driven

對應資料夾：sample-01-basic/（shipping.go + shipping_test.go）｜ 跑：go test ./013-testing/sample-01-basic

1. 三個雷打不動的慣例

Go 的測試不用裝任何套件、也沒有 assert 函式，全靠標準庫 testing ＋三條硬慣例：

慣例規則為什麼
檔名一定是 xxx_test.gogo test 只認這種檔名；正式 go build 時它會被忽略（不會進 binary）
函式簽名func TestXxx(t *testing.T)Test 大寫開頭 + 收一個 *testing.T，go test 才會把它當測試跑
package通常和被測程式同 package同 package 才測得到未匯出（小寫）的東西；shipping_test.go 就寫 package shipping
 
// shipping.go —— 被測程式：滿 1000 免運，否則 60
package shipping

func Fee(orderAmount int) int {
    if orderAmount >= 1000 {
        return 0
    }
    return 60
}
// shipping_test.go —— 最基本的測試：自己 if 比對，不符就 t.Errorf
package shipping

import "testing"

func TestFee(t *testing.T) {
    if got := Fee(1000); got != 0 {
        t.Errorf("Fee(1000) = %d, want %d", got, 0)   // 慣例訊息：got vs want
    }
}
失敗訊息的慣例寫法Go 圈子習慣寫成 函式(輸入) = 實際, want 期望，例如 Fee(1000) = 999, want 0。看 log 一眼就知道「誰、輸入什麼、拿到什麼、該是什麼」。
t.Errorf vs t.Fatalft.Errorf＝記一筆失敗但繼續往下跑（想一次看到多個錯用它）；t.Fatalf＝記失敗並立刻中止這個測試（用在「前置條件都不成立、再測下去沒意義」，例如某個回傳的 err != nil 就別再碰它的結果了）。都不會讓程式 panic，只影響「這個測試」。

2. table-driven：一張表 + 一個迴圈，涵蓋多案例

只測一兩個值不夠。Go 最常見的寫法是把「輸入 / 期望」列成一張表，再用迴圈跑，每列包一層 t.Run(子測試名, ...)：

func TestFee_Table(t *testing.T) {
    tests := []struct {
        name string   // 子測試名，會顯示在輸出
        in   int       // 輸入
        want int       // 期望
    }{
        {"剛好滿千免運", 1000, 0},
        {"差一元未達門檻", 999, 60},
        {"金額為零收運費", 0, 60},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {          // 每列一個子測試
            if got := Fee(tt.in); got != tt.want {
                t.Errorf("Fee(%d) = %d, want %d", tt.in, got, tt.want)
            }
        })
    }
}
為什麼 table-driven 好加案例＝加一列（不用複製一整個 test 函式）；配 t.Run 後，某列失敗會標出是哪一列（如 TestFee_Table/差一元未達門檻），而且其他列照跑不中斷，一次看到全部結果。
類比像 Excel 對答案：左邊填「題目」、右邊填「正解」，然後一列一列核。程式邏輯只寫一次，資料一直加就好。
t.Run 的名字子測試名會被正規化（空白轉底線）用在 -run 過濾，例如 go test -run TestFee_Table/金額為零 只跑那一列。取有意義的名字，log 才好讀。

3. 依賴抽成介面，測試時換成假實作

要測「註冊完寄歡迎信」的邏輯，但總不能每跑一次測試就真的寄一封信。解法＝把「怎麼送」抽成介面（呼應主題 007：介面在使用方定義、小介面單方法、隱式實作），測試時塞一個「假的實作」進去：

// notify.go —— 依賴的是「介面」，不是某個具體寄信器
package notify

type Notifier interface {
    Send(msg string) error
}

func Welcome(n Notifier, name string) error {   // 被測對象
    return n.Send("歡迎 " + name)
}
// notify_test.go —— mockNotifier：不真送，只記錄「被呼叫幾次 / 收到什麼」
package notify

import "testing"

type mockNotifier struct {
    calls   int
    lastMsg string
}

func (m *mockNotifier) Send(msg string) error {   // 有 Send → 自動實作 Notifier
    m.calls++
    m.lastMsg = msg
    return nil
}

func TestWelcome(t *testing.T) {
    mock := &mockNotifier{}
    if err := Welcome(mock, "小明"); err != nil {   // 前置條件壞了就直接中止
        t.Fatalf("Welcome 回傳非預期錯誤：%v", err)
    }
    if mock.calls != 1 {                            // 斷言①：剛好呼叫 1 次
        t.Errorf("Send 被呼叫 %d 次, want %d", mock.calls, 1)
    }
    if want := "歡迎 小明"; mock.lastMsg != want {    // 斷言②：訊息正確
        t.Errorf("Send 收到訊息 = %q, want %q", mock.lastMsg, want)
    }
}
mock 的本質＝實作介面mock 不是什麼魔法，就是「一個為測試而寫的假型別，剛好有介面要的方法」。它把互動（呼叫次數、收到的參數）記下來，讓你 斷言「被測程式有沒有正確使喚它」——不用碰真的資料庫 / 寄信服務。
為什麼能注入Welcome 收的是 Notifier（介面），不是 *EmailSender（具體型別）。正式跑傳真的寄信器、測試傳 mock——這就是「依賴注入 + 介面」讓程式好測的原因。
mock 命名可小寫因為 mockNotifier 只在測試檔用，故意小寫（未匯出）不外流。它一樣要用指標 receiver（*mockNotifier）才改得到自己的 calls；注入時記得傳 &mockNotifier{}。

概念為主（無對應單一 sample）；平常跑測試就這幾個。

4. 常用 go test 指令

指令做什麼
go test ./...跑整個 module 所有測試（最常用）
go test ./013-testing/...只跑某個目錄（含子目錄）底下的測試
go test ./013-testing/sample-01-basic跑單一 package 的測試
go test -run TestFee只跑名字符合的測試（正則）；子測試用 -run TestFee_Table/金額為零
go test -v ./...verbose，逐一印出每個測試 / 子測試 PASS/FAIL
go test -race ./...開競態偵測（並發程式必跑，呼應主題 010）
綠色才算過整包過會印 ok  golearn/013-testing/sample-01-basic；沒有測試檔會印 no test files；有一個 t.Errorf/t.Fatalf 觸發就整包 FAIL。第二次跑同樣結果會顯示 (cached)＝結果被快取、沒真的重跑。
驗證「測試有效」的小技巧把被測程式故意改壞（例如讓 Fee 回錯的值），跑 go test 確認它真的 FAIL、且訊息指得出哪裡錯，再改回來。一個「不管程式對錯都 PASS」的測試是假的安全感。

練習 動手練習（每題 ≤ 5 分鐘）

打開對應資料夾的 xxx_test.go，照 TODO 把測試補完，用 go test 驗證變綠。程式（被測那支）已經幫你寫好、可運作，你要練的是寫測試。寫完可請 Claude 批改、記進學習紀錄。

作業做什麼驗收
homework-01-basic/
（discount_test.go）		把 TestApply 改成 table-driven（比照 sample-01-basic 的 TestFee_Table），涵蓋多組折扣。補上並通過 Apply(1000,10)==900、Apply(1000,100)==0、Apply(1000,0)==1000；go test 全綠。
homework-02-mock/
（signup_test.go）		自己寫一個 fakeRepo 實作 Repo 介面，注入 Register，驗證 Save 有被呼叫到、且收到正確 id（比照 sample-02-mock）。Register(fake,"user-1") 後：err==nil、Save 呼叫 1 次、收到的 id 是 "user-1"；go test 全綠。
跑法對整個資料夾跑（帶點路徑）：go test ./013-testing/homework-01-basic；想看細節加 -v。注意：起始骨架本來就會 PASS（起始 case 或空測試），所以「綠」不代表做完——要照 TODO 真的補上斷言，才算把 homework 完成。
