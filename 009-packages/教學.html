A package、匯出與 import

對應檔案：sample-01-crosspkg/（真 package）｜ 跑：go run ./009-packages/sample-01-crosspkg

1. 一個資料夾 = 一個 package

Go 沒有「用檔案分模組」這回事——切 package 就是切資料夾。同一個資料夾裡所有 .go 檔開頭都寫同一個 package 名稱，它們合起來就是「一個 package」。慣例上資料夾名和 package 名取一致（資料夾 pricing/ → package pricing）。

// 009-packages/sample-01-crosspkg/pricing/pricing.go
package pricing

func Total(unitPrice, qty int) int { ... }   // 這個檔屬於 pricing 這個 package

2. 大小寫 = 唯一的存取控制（匯出 / 私有）

規則識別字（函式、型別、變數、struct 欄位、常數）大寫開頭＝匯出（public，跨 package 可用）；小寫開頭＝未匯出（private，只有同 package 能用）。Go 沒有 public/private 關鍵字，就靠大小寫。
名字身分別的 package 用得到嗎
pricing.Total匯出（大寫）✅ 可 pricing.Total(...)
pricing.applyTax未匯出（小寫）❌ 跨包呼叫會編譯失敗：cannot refer to unexported name

3. import 路徑 = module 名 + 資料夾路徑

要用別的 package，先 import 它的完整路徑（= go.mod 裡的 module 名 golearn ＋ 資料夾路徑），再用 套件名.函式名 呼叫：

// 009-packages/sample-01-crosspkg/main.go
package main

import (
    "fmt"
    "golearn/009-packages/sample-01-crosspkg/pricing"  // module 名 + 資料夾
)

func main() {
    fmt.Printf("含稅總額：%d 元\n", pricing.Total(590, 3))  // 套件名.函式名 → 1858
}
陷阱（我們踩過）：import 路徑只能用 ASCII！如果資料夾名有中文，go run ./資料夾 / go build 會直接報 malformed import path: invalid char '套'——因為資料夾名會變成 import 路徑的一部分，而 Go 的 import 路徑不收非 ASCII 字元。
這就是為什麼本主題（與 013 測試）這種「真 package」主題，資料夾用英文（009-packages）；而 001–008 那種單檔主題用 go run 檔.go（不經 import 路徑），資料夾才可以是中文。
怎麼跑真 package 用 go run ./009-packages/sample-01-crosspkg（帶點的路徑、對整個資料夾），不是 go run 某個.go。

B 電商專案的目錄結構慣例

概念為主（無對應單一 sample）；接手電商 repo 時最常看到的 layout。

4. cmd / internal / pkg

Go 沒有硬性規定目錄結構，但社群有一套「大家都這樣擺」的慣例（源自 standard-go-project-layout）。看電商 repo 幾乎都會遇到：

目錄放什麼重點
cmd/各個「進入點」main package，如 cmd/api/、cmd/worker/一個子資料夾一顆執行檔
internal/只給「本專案」用的私有 package（業務邏輯多半在這）Go 強制：internal/ 裡的 package 外部 module 無法 import
pkg/刻意想給「外部」也能重用的函式庫非必要；沒有要被外部用就別放
 
myshop/
├── go.mod                  // module github.com/acme/myshop
├── cmd/
│   └── api/main.go         // 進入點：啟動 HTTP server
├── internal/
│   ├── order/              // 訂單領域（外部 import 不到）
│   └── payment/            // 金流領域
└── pkg/
    └── money/              // 可對外重用的金額工具
internal/ 是語言層級的保護把不想被外部依賴的東西放進 internal/，編譯器會擋掉外部 module 的 import。電商核心邏輯通常都藏在這裡。

C 命名慣例與 fmt / vet

對應檔案：sample-02-命名慣例.go（單檔）｜ 跑：go run 009-packages/sample-02-命名慣例.go

5. Go 的命名慣例

情境慣例例
一般命名MixedCaps 駝峰，不用底線getUserName（非 get_user_name）
縮寫整段同大小寫OrderID、httpURL（非 OrderId/httpUrl）
常數也是 MixedCaps，不是 SCREAMING_CASEmaxRetry（非 MAX_RETRY）
error 變數ErrXxxErrNotFound（主題 008 已見）
package 名短、全小寫、無底線order、pricing
為什麼命名慣例讓「大小寫」不只是風格、而是語意（大寫＝匯出）；縮寫整段大寫則避免 OrderId 這種一半大一半小的尷尬。整個 Go 生態都這樣寫，照做才「像 Go」。

6. gofmt 與 go vet

gofmt：沒有風格之爭Go 有官方統一排版，存檔跑 gofmt -w 檔（或編輯器自動）就對齊，team 裡不用吵縮排/括號。本 repo 檢查格式：gofmt -l 資料夾/*.go（列出不合格式的檔，空＝OK）。
go vet：抓編譯器不管的地雷如 Printf 格式字串對不上參數、鎖被複製等。單檔／單 package 檢查：go vet 檔 或 go vet ./資料夾。
提醒go vet／gofmt 不會幫你改命名（那是 golint/revive 之類的事）；命名要自己照慣例來。

小抄 本主題慣例與陷阱總表

情境慣例 / 正解
切 package切資料夾；同資料夾同 package；資料夾名＝package 名
公開 / 私有大寫＝匯出、小寫＝私有（唯一存取控制）
跨包呼叫import "module名/資料夾" → 套件名.函式名
import 路徑只能 ASCII；真 package 主題資料夾用英文
私有邏輯放 internal/，外部 module import 不到
命名MixedCaps、不用底線、縮寫整段大寫、ErrXxx、package 名短小寫
格式 / 靜態檢查gofmt -w（無風格之爭）、go vet（抓地雷）
