A struct：定義電商 model、建立值、&T{} vs new

對應檔案：sample-01-struct.go ｜ 跑：go run 005-struct與方法/sample-01-struct.go

1. 定義 struct —— 把相關欄位綁成一個型別

struct 就是「一組欄位打包成的自訂型別」。電商裡的 Product、Order、Customer 都是 struct。欄位大寫開頭 = 匯出（別的 package 或 JSON 存取得到），小寫 = 私有。

type Product struct {
    ID    string
    Name  string
    Price int   // 單價（元）
    Stock int
}
陷阱：金額別用 float電商金額一律用 int 存「元」（或最小單位）。float64 算錢會有精度誤差（0.1 + 0.2 != 0.3），對帳會出事。這條是硬規則。

2. 建立 struct 值 —— 用「具名欄位」

三種寫法，實務上請用具名欄位（寫出欄位名）：順序可隨意、沒填的欄位自動補零值、加欄位也不會壞。

k := Product{ID: "P001", Name: "機械鍵盤", Price: 2490, Stock: 50}  // 具名（推薦）
m := Product{"P002", "無線滑鼠", 590, 120}                          // 位置（不推薦）
c := Product{ID: "P003", Name: "USB-C 線", Price: 190}             // 沒填 Stock → 自動 0
慣例建 struct 用具名欄位 T{Field: value}。位置寫法只在欄位極少、意義明顯時偶爾用；欄位一多就難讀、日後加欄位還會整包爆掉。用 %+v 印 struct 會連欄位名一起顯示，debug 很好用。

3. 存取欄位 & &T{} vs new(T)

用「點」存取或修改欄位（k.Stock -= 1）。建立時要「值」還是「指標」：

寫法拿到什麼能否順便填欄位
Product{...}一個 Product 值可
&Product{...}指標 *Product可（最常用）
new(Product)指標 *Product（全零值）不可，只能拿到空殼再填
 
p := &Product{ID: "P001", Name: "機械鍵盤"}  // 建立即拿指標，還能填欄位
fmt.Println(p.Name)                        // 指標照樣用點存取，不必寫 (*p).Name

b := new(Product)                          // 等同 &Product{}，但不能填欄位
b.Name = "待補商品"
為什麼 &T{} 最常見電商 code 建 model 常回傳指標：省一次結構複製、也方便讓「會改自身的方法」動到本體（見群組 B 第 6 節與主題 006）。實務上 &T{...} 幾乎取代了 new(T)——一行就把「配置 + 初始化」做完。

B 方法：把行為綁在 model 上（receiver）

對應檔案：sample-02-方法.go ｜ 跑：go run 005-struct與方法/sample-02-方法.go

4. 方法 = 帶 receiver 的函式

方法就是「綁在某個型別上的函式」。寫法是在 func 和函式名之間，多一個 (receiver)。o 就像其他語言的 this / self，但 Go 不用那種關鍵字——你自己命名它。

func (o Order) Total() int {      // 讀作「Order 有一個 Total 方法」
    return o.Product.Price * o.Qty
}

order.Total()                     // 用「值.方法()」呼叫
慣例receiver 名字用型別首字母小寫（Order → o、Product → p），而且不要取名叫 this／self。同一型別的所有方法，receiver 名字要一致。
類比函式是「放在櫃檯的通用工具」，方法是「掛在某個物件身上的技能」。Total() 掛在 Order 身上，所以你對「一張訂單」說「算你的總額」，很自然。

5. 方法可帶參數、有回傳值

方法跟一般函式一樣，能收參數、回傳結果。這裡 value receiver（(o Order)）代表呼叫時 o 是「這個 Order 的一份複製」，很適合只讀不改的計算。

func (o Order) TotalWithDiscount(percent int) int {
    total := o.Total()
    return total * (100 - percent) / 100   // 先乘再除，維持整數運算
}
order.TotalWithDiscount(10)                // 打 9 折

6. value receiver 改欄位「改不到本體」

用 value receiver 時，方法裡動到的是複製品，函式一結束就丟了，外面的 struct 沒變：

func (o Order) doubleQtyWrong() {
    o.Qty *= 2       // 只改到複製品，白改
}
order.doubleQtyWrong()
// order.Qty 仍是原值，沒變成兩倍
陷阱想讓方法「改得到本體」，receiver 要用指標：func (o *Order) ...。這是初學最常見的困惑，但「value vs pointer receiver 該用哪個」牽涉到指標，留到主題 006 講透——現在只要記得：要改自身就用 * receiver、純計算用值 receiver。

C struct tag：控制序列化出來的 JSON

對應檔案：sample-03-tag.go ｜ 跑：go run 005-struct與方法/sample-03-tag.go

7. tag 語法 —— 欄位後面接一段反引號字串

struct tag 是接在欄位後面的一段反引號字串，格式慣例 key:"value"。json 這個 key 就是給標準庫 encoding/json 看的，用來指定序列化出來的鍵名。

type Product struct {
    ID    string `json:"id"`      // 序列化成 "id"（不寫 tag 預設用 "ID"）
    Name  string `json:"name"`
    Price int    `json:"price"`
}
重點：tag 本身不做事tag 只是掛在欄位上的「附註字串」，語言本身不理它。是 encoding/json 這類套件用反射(reflection) 去讀它、才有效果。所以打錯 tag（例如拼成 jsonn）不會編譯錯，只會默默失效——要小心。完整 JSON 操作在主題 012。

8. 常用選項：omitempty 與 "-"

Discount int    `json:"discount,omitempty"`  // 值為 0（零值）時整個省略
Internal string `json:"-"`                    // "-" = 永遠不進出 JSON

上面的 Product 若 Discount 是 0、Internal 有值，轉出來的 JSON 會是：

{
  "id": "P001",
  "name": "機械鍵盤",
  "price": 2490
}   // discount 因零值被 omitempty 省略；Internal 因 "-" 不輸出

9. 陷阱：只有「大寫欄位」會被序列化

小寫（未匯出）欄位，encoding/json 從 package 外看不到，直接跳過——就算硬加 tag 也一樣被忽略（而且 go vet 會警告「有 tag 卻沒匯出」）。

costPrice int   // 小寫 c → JSON 完全看不到它（連 tag 都別加）
最常見的雷「我的欄位怎麼都沒進 JSON？」十之八九是欄位名是小寫。要被序列化的欄位一定要大寫開頭；反序列化（Unmarshal）也一樣，記得傳指標 &v 才填得進去。

D embedding：用組合共用欄位（不是繼承）

對應檔案：sample-04-embedding.go ｜ 跑：go run 005-struct與方法/sample-04-embedding.go

10. 嵌入(embedding) —— 匿名欄位

很多 model 都要「建立者 / 建立時間」這種稽核欄位。與其每個 struct 複製一份，不如抽成 Audit，用嵌入塞進去：在欄位列表寫一個只有型別名、沒有欄位名的欄位。

type Audit struct {
    CreatedBy string
    CreatedAt string
}
type Order struct {
    Audit          // ← 匿名（嵌入）：沒給欄位名，只寫型別
    ID    string
    Total int
}
對比 has-a群組 A 的 Product Product（具名欄位）是「Order 有一個 Product」；這裡 Audit（匿名）是把 Audit 的內容攤平嵌進 Order。差別就在「有沒有欄位名」。

11. 欄位/方法「晉升(promotion)」

嵌入之後，內層的欄位和方法會晉升到外層，可以像自己的一樣直接用：

order.CreatedAt        // 晉升欄位：明明定義在 Audit，卻能直接 order.CreatedAt
order.Summary()        // 晉升方法：Audit 的方法也能直接 order.Summary()
order.Audit.CreatedBy  // 仍可用完整路徑存取內層本體（指到同一個東西）

12. 這是「組合」，不是繼承

核心觀念：組合取代繼承Go 沒有繼承（沒有 extends、沒有父類別）。embedding 是組合(composition)：Order 是「有一個 Audit」，不是「是一種 Audit」。它只是把欄位/方法借過來用得順手，兩者仍是各自獨立的型別。
類比繼承像「基因遺傳」（你天生就是父類別的一種）；組合像「裝配零件」（訂單裝上一塊稽核模組）。Go 選了後者——想加功能就「裝一塊上去」，而不是「認一個爸」，搭配起來更自由、耦合更低。這也是 Go 社群常講的「composition over inheritance」。
