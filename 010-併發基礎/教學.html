你會學到怎麼用 go 開 goroutine 並行做事、用 sync.WaitGroup 等它們全部完成、用 channel 在 goroutine 之間傳資料、用 sync.Mutex 保護「大家會同時改的變數」、以及用 select 同時等多個 channel（含逾時）。四個 sample 各對應一組概念，都是安全、race-free 的正確寫法。
怎麼用這份教材每節標題下方會標「對應哪個 sample 檔」。先讀說明 → 打開該 sample 對照註解 → go run 跑一次看輸出（併發輸出順序可能不同，屬正常）→ 最後做動手練習。
一句話心法Go 的併發哲學是「不要用共享記憶體來通訊，要用通訊來共享記憶體」—— 能用 channel 讓資料流動，就別讓一堆 goroutine 去搶同一塊變數；真的非搶不可，才上 Mutex。

A goroutine：開背景並行，用 WaitGroup 等全部完成

對應檔案：sample-01-goroutine.go ｜ 跑：go run 010-併發基礎/sample-01-goroutine.go
核心觀念在任何函式呼叫前面加一個 go，那個呼叫就會被丟到背景並行執行，main 不會等它、直接往下跑。goroutine 極輕量（起一個大約幾 KB），開幾千個都沒問題 —— 這是 Go 併發好用的根本原因。

1. go：把一件事丟到背景

電商情境：一筆訂單成立，要同時寄 Email、SMS、App 推播。這三件事互相獨立又都在等網路，各開一個 goroutine 並行做最划算：

go func(channel string) {
    // 這段在背景跑；main 不會停下來等它
    sendNotify(channel)
}(ch)   // ← 立刻回來繼續下一圈，不阻塞
陷阱：main 結束，goroutine 就被收掉如果 main 沒等，它自己跑完就 return，程式結束，背景還沒做完的 goroutine 會被直接砍掉（通知可能一封都沒寄出）。所以幾乎每次開 goroutine，都要有辦法「等它做完」—— 那就是下一節的 WaitGroup。

2. sync.WaitGroup：等一組 goroutine 全部完成

WaitGroup 就是個計數器，用三個動作：Add(n) 加計數、Done() 減 1、Wait() 卡住直到歸零。心法：開幾個就 Add 幾個（且 Add 要在 go 之前）、每個 goroutine 收尾 Done、主流程 Wait。

var wg sync.WaitGroup
for _, ch := range channels {
    wg.Add(1)              // 先加計數（務必在 go 之前）
    go func(channel string) {
        defer wg.Done()    // 收尾減 1
        sendNotify(channel)
    }(ch)
}
wg.Wait()                  // 卡住，直到三個都 Done 才往下
fmt.Println("全部寄完")     // 保證在所有通知之後才印
陷阱：Add 要在 go 外面如果把 wg.Add(1) 寫到 goroutine 裡面，可能 Wait() 已經先跑到、計數還沒加上去，就提早通過了。正確順序是「先 Add、再 go」。

3. defer wg.Done()：保證計數一定會減

Done() 一定要配 defer（呼應主題 002 的 defer）：這樣就算 goroutine 中途 return 或 panic，計數照樣會 -1；否則 Wait() 會永遠等不到歸零 —— 程式卡死（deadlock）。

更現代的寫法（Go 1.25+）新版可用 wg.Go(func(){ ... })，它會自動幫你 Add(1) 並在函式結束時 Done()，計數不可能寫漏。本 sample 仍用經典的 Add/Done/Wait，因為你在既有電商 code 裡看到的絕大多數是這個寫法 —— 兩個都要看得懂。

B channel：goroutine 之間的輸送帶

對應檔案：sample-02-channel.go ｜ 跑：go run 010-併發基礎/sample-02-channel.go
類比channel 是一條有型別的輸送帶：一端把東西放上去（送），另一端拿下來（收）。goroutine 之間不用共享變數、不用鎖，資料直接在輸送帶上流動 —— 這就是「用通訊共享記憶體」。

4. 建立、送、收

make(chan T) 建一條只能流 T 的 channel；ch <- v 送、v := <-ch 收。預設的 channel 是無緩衝的：送的那一方會卡住，直到有另一方來收（這是天然的同步點）。

jobs := make(chan Order)   // 只能流 Order
results := make(chan int)   // 只能流 int

jobs <- order               // 送：沒人收就在這裡等
subtotal := <-results       // 收：沒東西就在這裡等
陷阱：無緩衝 channel 會互等成 deadlock如果在同一個 goroutine裡「先送、再收」同一條無緩衝 channel，送的時候就卡死了（沒有別人來收）。sample 裡因此用一個獨立 goroutine 負責送 jobs，主流程負責收 results，兩邊分開才不會互鎖。

5. close 與 for range：收到「關閉」為止

送方送完後 close(ch) 宣告「沒有更多東西了」；收方用 for v := range ch 一直收，channel 一被 close，這個迴圈就自然結束。這是「不知道會有幾筆、收到沒有為止」最乾淨的寫法。

// 送方：送完就關
for _, o := range orders { jobs <- o }
close(jobs)

// 收方：range 到 channel 關閉就停
for o := range jobs {
    process(o)
}
慣例只有送方 close，收方不要 close（也不要送已關閉的 channel，會 panic）。close 是「我這邊沒東西要送了」的訊號，交給負責送的那方發出最合理。

6. worker pool：多個 worker 分食同一條 jobs

把「起 N 個 worker、每個都 for range jobs」合起來，就是電商後端常見的 worker pool：一批訂單丟進 jobs，N 個 worker 並行搶著處理，結果丟回 results。sample 用 WaitGroup 等所有 worker 做完後再 close(results)，主流程才知道可以停止收集。

func worker(id int, jobs <-chan Order, results chan<- int) {
    for o := range jobs {          // 搶到一筆就處理
        results <- o.Price * o.Qty
    }
}
慣例：channel 的「方向型別」參數寫成 <-chan Order（只能收）、chan<- int（只能送），把用途寫進型別。這樣編譯器會擋掉「worker 不小心往 jobs 送」這種錯，也讓讀 code 的人一眼看懂資料流向。

C sync.Mutex：保護「大家會同時改」的共享狀態

對應檔案：sample-03-Mutex.go ｜ 跑：go run 010-併發基礎/sample-03-Mutex.go（或加 -race）
什麼時候需要 Mutex當多個 goroutine 會同時讀寫同一塊記憶體（例如一個庫存數字、一個計數器），就需要保護。channel 能解決的優先用 channel；但「大家要更新同一個共享變數」這種，用 Mutex 最直接。

7. data race：不保護會發生什麼

熱賣商品同時湧入下單，很多 goroutine 同時扣同一個 stock。「扣庫存」其實是三步：讀 stock → 減 1 → 寫回。兩個 goroutine 可能都讀到 100、各自算成 99、各自寫回 99 —— 賣了兩件卻只扣一件。這種「結果取決於誰先誰後」的錯就叫 data race，會超賣。

陷阱：data race 很隱蔽它不一定每次都出錯，可能測起來都對、上線量一大才偶爾少扣，最難抓。不要靠運氣：多個 goroutine 共享可變狀態，就一定要保護。

8. Lock + defer Unlock：把「讀-改-寫」圍成不可分割的一段

sync.Mutex（互斥鎖）保證同一時間只有一個 goroutine能進入 Lock() 與 Unlock() 之間那段。慣例：Lock 後立刻 defer Unlock，成對出現、就算中途 return 也一定會解鎖。

func (inv *Inventory) Deduct(n int) bool {
    inv.mu.Lock()
    defer inv.mu.Unlock()      // 一鎖就登記解鎖

    if inv.stock < n {
        return false           // 中途 return，defer 仍會 Unlock
    }
    inv.stock -= n             // 同一時間只有一個人在做這行
    return true
}
慣例：鎖和它保護的資料放同一個 struct把 mu sync.Mutex 和 stock int 放進同一個 Inventory，一看就知道「這把鎖保護誰」。Mutex 的零值就是「未上鎖」，不用初始化，直接用。讀共享資料（如 Stock()）只要有別人可能同時寫，也要上鎖。

9. 結果驗證：100 個並行扣款，精準到 0

sample 開 100 個 goroutine、每個扣 1 件，靠 Mutex 保護，最後剛好扣到 0。想證明「沒鎖真的會錯 / 有鎖真的沒事」，用 Go 內建的競態偵測器：

go run -race 010-併發基礎/sample-03-Mutex.go
神器：-race加上 -race 執行（或測試時 go test -race），Go 會在跑的時候幫你偵測有沒有「沒保護的並行讀寫」，抓到就印出 DATA RACE 和出事的兩行。本 sample 上了鎖，跑 -race 是乾淨的；把 Lock/Unlock 註解掉再跑，就會看到它報警。寫併發程式養成用 -race 的習慣。

D select：同時等多個 channel，誰先好就處理誰

對應檔案：sample-04-select.go ｜ 跑：go run 010-併發基礎/sample-04-select.go
類比select 之於 channel，就像 switch 之於值：它同時盯著好幾條 channel，哪一條先有動靜就走那個 case；若當下沒有任何一條就緒，就整個卡住等（除非有 default）。

10. select 多路等待：取最快回覆的

電商情境：向兩家供應商同時詢價，誰先回就用誰的報價。兩條 channel 一起等，select 會選先就緒的那個：

select {
case p := <-supplierA:
    fmt.Println("採用 A：", p)
case p := <-supplierB:
    fmt.Println("採用 B：", p)   // 兩者先就緒的勝出
}
陷阱：多個同時就緒是「隨機」挑一個如果 select 有兩個以上 case 同時就緒，Go 會隨機選一個（不是由上到下）。所以別依賴 case 的先後順序來決定優先權。

11. select + time.After：加逾時，別傻等

不能無限等一家慢供應商拖垮結帳。time.After(d) 回傳一條 channel，它會在 d 之後才送出值 —— 拿它當一個 case，就是「逾時」分支：

select {
case p := <-slow:
    use(p)
case <-time.After(100 * time.Millisecond):
    fmt.Println("詢價逾時，改用預設價")   // 100ms 到了還沒回就走這
}
前導：這就是 011 context 的雛形「等一件事，但最多等這麼久，逾時就放棄」是電商後端每天在做的事。這裡用 select + time.After 手做一次；下個主題 011 context 會把「逾時 / 取消」包裝成標準的 context.WithTimeout，配 ctx.Done()（也是一條 channel）搭 select 用。

12. select + default：非阻塞地「看一眼」

加一個 default，select 就不阻塞：如果沒有任何 case 當下就緒，立刻走 default，繼續做別的事。適合「探看一下 channel 有沒有東西，沒有就算了」：

select {
case msg := <-stockReport:
    handle(msg)
default:
    // 沒有回報就別等，往下做別的
    fmt.Println("目前沒有庫存回報")
}
慣例要「阻塞等到有東西」就不寫 default；要「有就拿、沒有就跳過」就加 default。這跟 002 那個「switch 也可以有 default」的直覺一致。
