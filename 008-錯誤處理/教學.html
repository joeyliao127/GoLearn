主題 008 · 錯誤處理

Go 沒有 try/catch——error 是一個「回傳值」，用 if err != nil 早早處理掉。學會回傳/包裝/比對錯誤，以及 panic/recover 的少數時機，就能讀懂電商 repo 裡結帳、庫存、金流那一長串 return ..., err 到底在做什麼。

A error 基礎：error 是值、(值, error) 慣例、早 return

對應檔案：sample-01-error基礎.go ｜ 跑：go run 008-錯誤處理/sample-01-error基礎.go

1. Go 沒有 try/catch —— error 就是一個「值」

別的語言遇到錯誤是「丟例外」（throw / try...catch）；Go 完全不這樣。Go 把錯誤設計成一個普通的回傳值，型別是內建的 error 介面。函式覺得出事了，就「多回傳一個 error 給你」，由你決定怎麼辦。

慣例：error 放在回傳值的最後一個，長相固定是 func f(...) (結果, error)。error 是 nil 代表「沒事」；非 nil 代表「出事了，內容看這個錯誤」。

func withdraw(balance, amount int) (int, error) {
    // ...檢查...
    return balance - amount, nil   // 成功：error 位置給 nil
}
心法反轉在 Go，錯誤不是「意外的中斷」，而是資料流的一部分。你會一直看到 x, err := f()——把「結果」和「可能的錯誤」一起接回來，這是整個 Go 生態的地基。

2. 建立錯誤：errors.New vs fmt.Errorf

兩種常見做法，差別只在「訊息要不要塞變數」：

用哪個時機例子
errors.New("...")訊息是固定字串errors.New("扣款金額必須大於 0")
fmt.Errorf("...", ...)訊息要帶變數（像 Printf）fmt.Errorf("餘額不足：欲扣 %d，只有 %d", a, b)
訊息慣例錯誤字串小寫開頭、結尾不加標點（如 "庫存不足" 不是 "庫存不足。"）。原因：錯誤常被「串起來」印出（結帳失敗: 庫存不足），大寫或句點接起來會很醜。

3. 呼叫方慣例：if err != nil 早 return

拿到 (值, err) 後，先檢查 err：有問題就立刻收尾（回傳、記 log、結束），正常邏輯留在後面。這叫 early return（早 return）——把錯誤路徑先擋掉，主線就能平鋪直敘、不必一路往右縮排。

newBalance, err := withdraw(balance, 300)
if err != nil {
    fmt.Println("扣款失敗：", err)
    return                 // 錯了就早早收尾
}
// 走到這裡，保證成功，安心用 newBalance
fmt.Printf("餘額 %d 元\n", newBalance)
類比像過安檢：不合格的當場攔下請回（early return），後面走廊只剩「已通過」的人，動線清爽。若改成「層層 else 巢狀」，就像把所有人塞進一條不斷分岔的隊伍，越讀越縮排、越難跟。
陷阱：別吞錯最忌 x, _ := f() 把 err 用 _ 丟掉、假裝沒事——出錯時你會完全不知情，還拿著半殘的值繼續跑。除非你真的確定不可能錯，否則每個 err 都要處理（至少往上回傳）。

B 包裝與比對：%w 包裝、errors.Is

對應檔案：sample-02-包裝與Is.go ｜ 跑：go run 008-錯誤處理/sample-02-包裝與Is.go

4. 用 %w 包裝：補上下文，又不丟掉原始錯誤

錯誤往上傳時，常想順手加一句「是誰、在哪出事」。fmt.Errorf 有個特別的動詞 %w（wrap，包裝）：它把原始 err 藏進新錯誤裡，之後還挖得出來。

if err := checkStock(productID, qty); err != nil {
    return fmt.Errorf("訂單 %s 建立失敗: %w", orderID, err)
    //                               ^^ 用 %w 把 err 包進去
}
%w 和 %v 差在哪%v 只是把錯誤的文字拼進去，原始錯誤的「身分」就斷了；%w 會保留那條「錯誤鏈」，讓下面第 6 節的 errors.Is／第 9 節的 errors.As 還能穿透比對。想被 Is/As 認出來，就得用 %w。

5. 包裝後：別再用 == 比對

一旦包過，錯誤就換了一層「外皮」，直接 err == ErrOutOfStock 會是 false（比的是最外層那顆，不是裡面包的）：

err := placeOrder("ORD-0001", "B200", 2)   // 內部缺貨，被包了一層
fmt.Println(err == ErrOutOfStock)          // false！比的是外皮
陷阱看到「用 == 比對錯誤卻老是不成立」，八成是對方被 %w 包過了。解法不是拆包裝，而是改用 errors.Is（見下）。

6. errors.Is：穿透整條鏈比對「是不是某個錯誤」

errors.Is(err, 目標) 會沿著 %w 一層層往裡剝，只要鏈上任一層等於目標就回 true。這才是比對 sentinel error 的正解：

if errors.Is(err, ErrOutOfStock) {
    // 不管被包了幾層，只要底下是缺貨就進來
    fmt.Println("顯示『到貨通知我』按鈕")
}
慣例比對錯誤「是不是某個已知值」，一律用 errors.Is，不要用 ==——因為你永遠不知道中間會不會有人包了一層。（想手動剝一層看裡面是誰，可以用 errors.Unwrap，但實務多半直接 Is。）

C sentinel 與自訂型別：ErrXxx、自訂 error、errors.As

對應檔案：sample-03-sentinel與自訂.go ｜ 跑：go run 008-錯誤處理/sample-03-sentinel與自訂.go

7. sentinel error：可被比對的「固定錯誤值」

當一個錯誤「只需要被認出來，不必夾帶額外資料」（例如「付款被拒」就是有或沒有），就用 sentinel error：一個套件層級變數，慣例名字 ErrXxx，用 errors.New 建立。

var ErrOutOfStock     = errors.New("庫存不足")
var ErrPaymentDeclined = errors.New("付款被拒")
慣例錯誤變數一律 Err 開頭大駝峰（ErrNotFound）。標準庫也一樣：io.EOF、sql.ErrNoRows、fs.ErrNotExist——你在電商 repo 裡會常常 errors.Is(err, sql.ErrNoRows) 判斷「查無資料」。

8. 自訂 error 型別：需要「夾帶資料」時

固定字串不夠用時（例如想告訴呼叫方「找不到的是哪種資源、哪個 ID」），就自訂一個 error 型別：一個 struct，實作 Error() string 方法就自動是合法的 error（error 是介面，隱式實作、不用寫 implements）。

type NotFoundError struct {
    Resource string   // 帶資料：是哪種資源
    ID       string   // 帶資料：哪個 ID
}
func (e *NotFoundError) Error() string {
    return fmt.Sprintf("找不到%s：%s", e.Resource, e.ID)
}
類比sentinel 像「缺貨」這張現成的號碼牌——大家比對同一張；自訂 error 型別像一張可填欄位的表單，能寫上「缺的是哪個商品、缺幾個」，資訊量大得多。

9. errors.As：把「某個型別」從鏈裡挖出來

errors.Is 回答「是不是某個值」；errors.As 回答「鏈上有沒有某個型別，有就把它還原成具體型別給我」——這樣才能讀它的欄位。用法：宣告一個目標指標變數，把它的位址（&nfErr）交給 As。

var nfErr *NotFoundError
if errors.As(err, &nfErr) {          // 注意傳的是 &nfErr（位址）
    fmt.Println(nfErr.Resource, nfErr.ID)   // 還原成型別，讀欄位
}
一句話分工errors.Is ＝「這是不是那顆（sentinel）？」；errors.As ＝「鏈裡有沒有那種（型別），撈出來讓我讀細節」。兩者都會穿透 %w 包裝。
陷阱：As 的第二參數一定是「指標的位址」要傳 &nfErr（nfErr 本身是 *NotFoundError，取址後是 **NotFoundError）。傳錯（例如只給 nfErr）會直接 panic。記法：As 要「寫進」你的變數，所以得給它變數的位址。

D panic 與 recover：少用的最後手段

對應檔案：sample-04-panic與recover.go ｜ 跑：go run 008-錯誤處理/sample-04-panic與recover.go

10. panic 的時機：真正的異常，不是一般錯誤流程

panic 會中斷正常流程、一路往上炸，沿途執行各層的 defer，若沒人攔就整個程式崩潰。它只該用在「這根本不該發生」的情況——程式的不變式被打破、或初始化時就掛（設定檔壞、必要環境變數缺）。

func divShare(amount, n int) int {
    if n == 0 {
        panic("divShare: 人數不可為 0（呼叫端邏輯有誤）")
    }
    return amount / n
}
判斷準則「可預期、會發生的失敗」（缺貨、餘額不足、查無資料、使用者輸入錯）→ 回 error；「照理說絕不會發生、發生就是 bug」→ 才考慮 panic。日常業務邏輯裡，你寫 panic 的次數應該接近零。

11. recover：只有在 defer 裡才有效

recover() 能攔住 panic、阻止程式崩潰，但它只有寫在 defer 的函式裡才作用（正常結束時 recover() 回 nil；發生 panic 時回 panic 的值）。常見用法是把 panic「降級」成一個普通 error 回傳：

func settleOne(orderID string, amount, n int) (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("結算 %s 發生嚴重錯誤: %v", orderID, r)
        }
    }()
    share := divShare(amount, n)   // 這裡 panic 會被上面 defer 接住
    // ...
    return nil
}
慣例用 recover 的地方通常搭配具名回傳值（(err error)），才能在 defer 裡把攔到的東西塞回 err。真實世界最常見在「每個 request 一層 middleware」攔住 handler 的意外 panic，讓伺服器回 500 而不是整個掛掉、拖垮其他請求。

12. 少用慣例：能回 error 就別 panic

把上面兩節收成一句：對外的 API、日常業務邏輯，一律用 error；panic/recover 是最後手段。濫用 panic 當「跨函式的 try/catch」是反模式——會讓錯誤流程變得不可預測、難以追蹤。

類比panic 像拉火災警報器：真失火（不變式崩了）才拉；不會因為「找不到某商品」這種日常小事就拉警報、疏散整棟樓。日常的「東西沒找到」，好好回一個 error 就夠了。
陷阱：別把 recover 當萬用 try/catch看到有人到處 defer recover() 把所有 panic 悶掉、當一般錯誤處理，要警覺——這會把「真正的 bug」也一起吞掉、藏起問題。recover 只該攔「你預期可能 panic 的那一小段」，攔完要嘛降級成 error、要嘛記 log 後重新 panic。

小抄 本主題慣例與陷阱總表

情境慣例 / 正解
函式可能出錯回傳 (結果, error)，error 放最後；成功時 error 給 nil
