A 介面基礎：隱式實作、duck typing、小介面

對應檔案：sample-01-介面基礎.go ｜ 跑：go run 007-介面/sample-01-介面基礎.go

1. 介面是什麼——一組「方法簽章」的集合

介面（interface）就是一張「能力清單」：列出「要有哪些方法」，但不寫實作。以「發訂單通知」為例，我們不在乎是 Email 還是簡訊，只要求「會 Send」：

type Notifier interface {
    Send(message string) error   // 只有簽章，沒有 { 實作 }
}
類比介面像徵才的「職位需求」：職缺寫「需要會 Send（發送）的人」，至於你是用 Email 部門還是簡訊部門的方式完成，公司不管——會做這件事就能上工。

2. 隱式實作——不用寫 implements

這是 Go 和 Java/C# 最不一樣的地方：一個型別只要「剛好」有介面要求的所有方法，就自動算實作了那個介面，完全不用宣告。

type EmailNotifier struct{ From string }

// 簽章 (message string) error 和 Notifier 完全一致 → 自動就是 Notifier
func (e EmailNotifier) Send(message string) error {
    fmt.Printf("[Email] %s\n", message)
    return nil
}
// 注意：整份檔案「沒有」任何 implements Notifier 的字樣
為什麼重要「實作」和「介面」可以互不知道對方存在：EmailNotifier 不必 import 定義 Notifier 的套件，也能滿足它。這讓你能替別人寫的型別（甚至標準庫的型別）補上「它符合我這個介面」，是 Go 解耦的關鍵。
怎麼確定有沒有實作到編譯器會在你「把它當介面用」時幫你檢查。想在編譯期就強制驗證，可寫一行 var _ Notifier = EmailNotifier{}——沒實作完就編譯失敗（這是社群常見的守門寫法）。

3. duck typing——「會這招，就當你是」

因為是隱式實作，Email 和 SMS 這兩個八竿子打不著的型別，都能塞進同一個 Notifier 變數。這種「不看你是什麼型別，只看你會不會這些方法」的風格叫 duck typing：

var n Notifier
n = EmailNotifier{From: "shop@x.com"}   // Email 當 Notifier
n.Send("訂單 #1001 已成立")

n = SMSNotifier{Gateway: "twilio"}       // 換 SMS，呼叫端一行都不改
n.Send("訂單 #1001 已出貨")

// 甚至能放進同一個 slice 一起處理：
channels := []Notifier{EmailNotifier{...}, SMSNotifier{...}}
for _, c := range channels { c.Send("週年慶 5 折起！") }
類比 duck typing「如果牠走路像鴨子、叫聲像鴨子，那就當牠是鴨子。」我們不驗血統（型別），只驗行為（方法）。

4. 小介面與 -er 命名慣例

Go 社群偏好小介面——常常只有一個方法。介面愈小，愈多型別能滿足它、愈好組合與 mock。標準庫的 io.Reader（只有 Read）、io.Writer（只有 Write）、fmt.Stringer（只有 String）都是範例。

慣例說明
單方法介面用 -er 命名方法叫 Read → 介面叫 Reader；Send → Notifier（動詞名詞化）。
介面要小「一個介面該有幾個方法？」答案通常是「愈少愈好」。需要多能力時用「組合多個小介面」而不是塞一個大介面。
用 %T 看真面目介面變數印 %T 會顯示底下「真正的動態型別」（如 main.EmailNotifier），除錯時很好用。
慣例先別急著設計大介面。實務上常是「先有具體型別，等發現有第二種實作、或需要 mock 時，才把共同的方法抽成一個小介面」——介面是需要時才長出來的，不是一開始硬湊的。

B 依賴注入（DI）：介面在使用方定義、實作可替換

對應檔案：sample-02-依賴注入.go ｜ 跑：go run 007-介面/sample-02-依賴注入.go

5. 介面「在使用方定義」——這是 Go 的關鍵慣例

電商後端常見分層：service（業務邏輯）需要 repository（資料存取）。Go 的慣例是：介面定義在「用它的那一方」（consumer，這裡是 service），而不是實作那一方。service 宣告「我需要能 Save / FindByID 的東西」：

// 定義在 service 這邊：這是 service 對「資料層」的最小需求
type OrderRepository interface {
    Save(o Order) error
    FindByID(id int) (Order, bool)
}
為什麼是使用方定義因為「需要什麼能力」是使用者才知道的事。這樣 service 只依賴自己宣告的小介面，跟任何具體資料庫實作零耦合；同一個 DB 實作也能同時滿足好幾個 service 各自的小介面。這跟 Java「先定義大介面、實作去 implement」的方向剛好相反。

6. 依賴注入——service 只認介面，實作從外面塞進來

OrderService 的欄位型別是介面，不是任何具體型別。它自己不 new 資料庫，而是等外部把實作「注入」進來（慣例是用一個建構函式 NewXxx 收介面參數）：

type OrderService struct {
    repo OrderRepository        // 欄位是「介面」，不是具體型別
}

func NewOrderService(repo OrderRepository) *OrderService {
    return &OrderService{repo: repo}   // 依賴從參數注入
}

func (s *OrderService) PlaceOrder(o Order) error {
    if o.Amount <= 0 {
        return fmt.Errorf("金額必須大於 0，收到 %d", o.Amount)
    }
    return s.repo.Save(o)       // 只透過介面方法做事
}
類比DI 像「插座與插頭」：OrderService 提供一個「插座」（OrderRepository 介面），記憶體版、MySQL 版、測試 mock 版都是不同「插頭」，插上哪個就用哪個，插座本身（service）完全不用改。

7. 換實作——呼叫端一行都不用改

記憶體版 repo「剛好」有 Save/FindByID，隱式滿足介面，可以直接注入。之後要換成真資料庫，只改「組裝那一行」，service 內部完全不動：

repo := NewInMemoryOrderRepo()      // ← 未來換成 NewMySQLOrderRepo() 就好
service := NewOrderService(repo)     // service 完全不知道背後是誰
service.PlaceOrder(Order{ID: 1001, Amount: 1500})
慣例這就是「面向介面編程」帶來的可替換性：組裝（依賴接線）集中在程式邊界（如 main 或啟動流程），業務邏輯只依賴介面。看電商專案時，留意「誰在 main 把實作接上 service」，那裡就是整個依賴圖的入口。

C mock：用假實作測 service、any/斷言、nil interface 陷阱

對應檔案：sample-03-mock.go ｜ 跑：go run 007-介面/sample-03-mock.go

8. mock——為「測試」而生的假實作

因為 service 只認介面，測試時我們可以塞一個假的（mock）實作，而不用真的連資料庫。mock 一樣靠「實作介面」，但它的工作是：①記錄互動（被呼叫幾次、傳了什麼）、②可被指定故意失敗（測 service 的錯誤處理）。

type MockOrderRepo struct {
    SaveCalls int       // 記錄 Save 被呼叫幾次
    LastSaved Order     // 記錄最後存了什麼
    FailWith  error     // 不為 nil 就讓 Save 回這個錯
}
func (m *MockOrderRepo) Save(o Order) error {
    m.SaveCalls++
    m.LastSaved = o
    if m.FailWith != nil { return m.FailWith }
    return nil
}

// 用 mock 驗證「正常下單」有沒有正確呼叫 Save：
mock := &MockOrderRepo{}
svc  := &OrderService{repo: mock}    // 注入 mock，不是真 DB
svc.PlaceOrder(Order{ID: 1001, Amount: 1500})
// 手動「斷言」：SaveCalls 應為 1、LastSaved 應為那筆訂單
為什麼能這樣測正因為 PlaceOrder 依賴的是介面而非具體 DB，我們才能無痛換上 mock。「金額為 0 時 Save 應被呼叫 0 次」這種互動驗證，就是靠 mock 記錄次數達成——這正是主題 013 用 testing 套件要自動化的事，這裡先手動體會。
慣例：mock 用指標mock 要「記錄狀態」（次數、內容），方法得用指標 receiver（func (m *MockOrderRepo)）才改得到自己的欄位，注入時也用 &MockOrderRepo{}。這呼應主題 006：要改到自身狀態就用指標 receiver。

9. 補充：any 與型別斷言

any 是 interface{}（空介面）的別名——「沒有任何方法要求」的介面，所以任何型別都滿足它，可裝任意值。要從 any 取回具體型別，用型別斷言，務必用「雙回傳值」版避免 panic：

var x any = "SAVE50"            // any 可裝任何東西

s, ok := x.(string)             // 雙回傳值：ok=true, s="SAVE50"（安全）
n, ok := x.(int)                // ok=false, n=0（不會 panic）

// 單回傳值版斷言失敗會 panic，少用：
// s := x.(int)   // ❌ panic: interface conversion
陷阱別濫用 any——它等於放棄型別檢查，把錯誤延到執行期。優先用「有方法的具體介面」；只有真的要裝「任意型別」（如解析未知 JSON）才用 any，且斷言一律用 v, ok := x.(T) 兩回傳值版。（多型別分支可用 switch v := x.(type)，主題 012 解析 JSON 時會再遇到。）

10. 陷阱：nil interface ≠ nil

Go 面試最愛考、實務也最容易中的雷。介面值內部其實有兩格：(型別, 值)。只有兩格都空時，介面才等於 nil。把一個「型別已知、但值為 nil 的指標」塞進介面，介面就不是 nil 了：

var repo OrderRepository        // 型別、值都空 → 真正的 nil
fmt.Println(repo == nil)        // true

var p *MockOrderRepo = nil      // 一個「值為 nil」的具體指標
repo = p                        // 介面現在帶著型別 *MockOrderRepo、值 nil
fmt.Println(repo == nil)        // false ！（型別那格不空）
陷阱最常見的翻車場景：函式回傳型別寫成介面（如 error），內部卻 return 一個「值為 nil 的具體指標」，外面 if err != nil 就會誤判成有錯。心法：要回 nil 介面，就直接 return nil，別回一個 nil 的具體指標再讓它被裝進介面。（008 錯誤處理會再提醒這點。）
